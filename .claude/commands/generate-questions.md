あなたは北米ソフトウェアエンジニア技術面接の問題作成の専門家です。
以下の指示に従って、指定されたトピックの面接問題を生成してください。

## 入力

$ARGUMENTS

（形式: `トピック名: 補足説明`）

## トピック特定

まず `prisma/seed.ts` を読み、`topics` 配列から入力にマッチするトピックの `id`, `name_en`, `name_ja`, `category_id` を特定してください。
英語名・日本語名どちらでもマッチ可能です。

**既存トピックにマッチしない場合**: 新規トピックとして扱います。

1. ユーザーにカテゴリ（既存 or 新規）を確認
2. `topic_id` は新しいUUIDを生成（`b1b2c3d4-0001-4000-8000-` の連番パターンを踏襲）
3. JSONファイルの `topic_id` にその新UUIDを設定し、`topic_name` にトピック英語名を記載
4. 「このトピックは新規です。DB投入前に `prisma/seed.ts` にトピックを追加して `pnpm db:seed` を実行してください」と案内

## 既存問題の重複チェック

トピック特定後、`questions/manifests/{slug}.md` を確認してください（存在する場合）。
このファイルには既存問題の一覧（タイプ・難易度・問題文の要約）が記載されています。

1. マニフェストを読み、新しく生成する予定の問題と**意味的に重複**するものがないか確認する
2. 重複の疑いがある場合、マニフェストに記載されたJSONファイル名を参照し、実際の問題ファイル（`questions/{filename}`）を読んで詳細を確認する
3. 重複が確認された場合、その問題はスキップし、別の切り口の問題を生成する
4. マニフェストが存在しない場合（新規トピック）、このステップはスキップする

**重複の判断基準**:

- 同じ概念を同じ角度から問うている → 重複（スキップ）
- 同じ概念でも異なる角度（例: 理論 vs コード、基礎 vs エッジケース）→ OK

## 問題の質の方針

- **「なぜそう動くのか」「内部で何が起きているのか」を問う問題**を作ること
- 知識の確認であっても、**具体的なシナリオや判断を伴う形式**にすること
- よくある誤解を突く問題、エッジケースを扱う問題が効果的
- コード実装問題（コードを書かせる）はアプリの目的外なので**生成しない**
- code（コード読解）問題は、表面的な出力予測ではなく、裏側のメカニズム理解を問う形にする
- explain（口頭説明）問題はメンタルモデル構築に有効なので積極的に使う

### NG/OK例（この基準で全問題を判定すること）

| NG（4択で定義を選ばせる）                          | OK（理解・判断を問う）                                            |
| -------------------------------------------------- | ----------------------------------------------------------------- |
| 「クロージャとは何ですか？」→ 定義を4択            | 「このコードでcountが保持される理由は？」→ メカニズムを4択        |
| 「let, const, varのスコープの違いは？」→ 事実を4択 | 「このforループでvarをletに変えると出力がどう変わるか？なぜか？」 |
| 「bind(), call(), apply()の違いは？」→ 説明を4択   | 「このコードで.bind(obj)を外すとthisはどうなるか？なぜか？」      |

※ 「クロージャとは何か説明せよ」は **explain型なら OK**（面接で実際に口頭で問われるため）

**簡易判定**: 問題文をGoogle検索して最初の結果で正解できる4択 → NG

## 問題タイプと構成ルール

- **multiple**（4択）: 概念の深い理解確認。**常に含める、全体の40-60%**
- **code**（コード読解）: 出力予測 + なぜその動作になるかの理解。**コード例が自然なトピックのみ**。純粋な理論トピックには使わない
- **truefalse**（○×判定）: よくある誤解・落とし穴チェック用。**最大4問**
- **explain**（口頭説明）: ルーブリック付き。**面接頻出トピックで1-3問**

### 選択肢（distractor）の品質基準

- **不正解選択肢はよくある誤解・部分的理解を反映すること**。明らかに無関係な選択肢は禁止
- 各不正解が「なぜ間違いか」を解説で説明できるレベルの妥当性を持つこと
- 正解と不正解の**文の長さ・具体性を揃える**（正解だけが長く詳細、は避ける）
- code問題: 不正解の出力値は、特定の誤解（例: ホイスティング無視、参照 vs コピーの混同）をした場合に得られる値にする

## ルール

- **総数**: 3-5問。量より質を優先。足りなければ追加生成する
- **難易度比率**: easy 40% / medium 50% / hard 10%
- **面接で出題頻度が高いものを選ぶこと**
- 各問題は **日本語・英語の両方** を含むこと
- 解説は「なぜ」を丁寧にわかりやすく説明すること。初学者でも理解できるよう、段階的に論理を積み上げる書き方にする。抽象的な説明だけでなく、具体例や比喩を交えて直感的に理解できるようにする
- 解説ではMarkdown記法を活用すること:
  - `インラインコード` で変数名・メソッド名・キーワードを囲む
  - **太字** で重要な概念を強調する
  - 必要に応じて短いコード例を ` ```js ` コードブロックで含める（5行以内目安、解説の補助として）
  - 改行で論理的な段落を分ける
- 北米SWE技術面接で実際に問われる内容を中心に。面接官は「説明してください」ではなく「このコードの問題点は？」「なぜこのアプローチが良いのか？」と聞く。その視点で問題を作る

## ワークフロー（4段階）

### Step 1: Markdownレビュー用ファイル生成

トピック特定後、まず `questions/{slug}_{YYYYMMDD}_{seq}.md` にMarkdown形式でレビュー用ファイルを作成する。

- `{slug}`: トピック名のスラッグ（例: "Closures & Scope" → "closures-scope"）
- `{YYYYMMDD}`: 生成日
- `{seq}`: 同一トピック・同日の連番（001, 002, ...）。既存ファイルを確認して衝突しないようにする

Markdownの形式:

```markdown
# {トピック英語名} — 問題レビュー

- **Topic ID**: {UUID}
- **Total**: {問題数}
- **Difficulty**: easy {n} / medium {n} / hard {n}
- **Types**: multiple {n} / code {n} / truefalse {n} / explain {n}

---

## Q1. [multiple | code | truefalse | explain] [easy | medium | hard]

**JA**: 問題文（日本語）

**EN**: Question text (English)

**選択肢** (multiple/code の場合):

- A: ...
- B: ...
- C: ...
- D: ...

**正解**: A (correct_index: 0) / true / false

**解説 JA**: ...

**解説 EN**: ...

**ルーブリック** (explain の場合):
| Score | JA | EN |
|-------|----|----|
| 1 | ... | ... |
| 2 | ... | ... |
| 3 | ... | ... |
| 4 | ... | ... |

**模範解答 JA**: ...

**模範解答 EN**: ...

---

## Q2. ...
```

#### 生成後セルフチェック（全問に適用）

Markdownを出力する前に、各問題を以下の基準で検証し、不合格なら作り直すこと:

1. **Google検索テスト**: 問題文をそのまま検索して即答できる4択 → 作り直し
2. **「なぜ」の存在**: 問題が「何か」だけでなく「なぜか」「どう動くか」を問うているか
3. **選択肢の妥当性**: 不正解選択肢が実際の誤解を反映しているか（明らかなダミーがないか）
4. **重複チェック**: 同じ概念を別の言い回しで繰り返していないか
5. **型と内容の一致**: 定義を問う内容が4択になっていないか（→ explain型に変更を検討）

生成後、ユーザーにレビューを依頼する。

### Step 2: レビュー・修正

ユーザーのフィードバックに基づいてMarkdownファイルを修正する。
承認（「OK」「LGTM」「いいよ」等）が得られるまでこのステップを繰り返す。

### Step 3: JSON生成 → DB投入

レビュー承認後:

1. Markdownから `questions/{slug}_{YYYYMMDD}_{seq}.json` を生成する（同名でmdをjsonに変えたファイル名）
2. `pnpm db:import questions/{slug}_{YYYYMMDD}_{seq}.json` を実行してdev環境に投入する
3. import成功後、スクリプトが自動的に `imported_dev: true` に更新する
4. prod投入はユーザーの指示があった場合のみ `pnpm db:import:prod` で実行する

### Step 4: マニフェスト更新

JSON生成後、`questions/manifests/{slug}.md` を更新する:

1. ファイルが存在しない場合は新規作成する（`questions/manifests/` ディレクトリも必要に応じて作成）
2. 新しく生成したJSONファイルのセクションを末尾に追加する
3. 各問題の要約行を `- [type/difficulty] question_enの要約` の形式で追記する
   - code問題やコードを含む問題は、コードブロック前のテキスト + コードの最初の行を含める
4. ヘッダーの `Total:` の問題数を更新する

※ マニフェストがずれた場合は `pnpm manifest:generate` で全体を再生成できる

## JSON出力形式

```json
{
  "topic_id": "<UUID>",
  "topic_name": "<トピック英語名>",
  "created_at": "<YYYY-MM-DD>",
  "imported_dev": false,
  "imported_prod": false,
  "questions": [
    {
      "type": "multiple | code | truefalse | explain",
      "difficulty": "easy | medium | hard",
      "question_ja": "...",
      "question_en": "...",
      "options": "<タイプに応じた形式>",
      "answer": "<タイプに応じた形式>",
      "explanation_ja": "...",
      "explanation_en": "..."
    }
  ]
}
```

### 各タイプの options / answer 形式

#### multiple / code

```json
{
  "options": [
    { "label": "A", "text_ja": "...", "text_en": "..." },
    { "label": "B", "text_ja": "...", "text_en": "..." },
    { "label": "C", "text_ja": "...", "text_en": "..." },
    { "label": "D", "text_ja": "...", "text_en": "..." }
  ],
  "answer": { "correct_index": 0 }
}
```

- options は必ず4つ
- correct_index は 0-3

#### truefalse

```json
{
  "options": null,
  "answer": { "correct_value": true }
}
```

#### explain

```json
{
  "options": {
    "rubric_ja": { "1": "...", "2": "...", "3": "...", "4": "..." },
    "rubric_en": { "1": "...", "2": "...", "3": "...", "4": "..." }
  },
  "answer": {
    "model_answer_ja": "...",
    "model_answer_en": "..."
  }
}
```

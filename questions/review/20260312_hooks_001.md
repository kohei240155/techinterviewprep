## Q1. [multiple] [easy]

**JA**: 以下のReactコンポーネントで、ボタンを1回クリックした後の `count` の値はいくつになりますか？

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  };

  return <button onClick={handleClick}>{count}</button>;
}
```

**EN**: In the following React component, what is the value of `count` after clicking the button once?

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  };

  return <button onClick={handleClick}>{count}</button>;
}
```

**選択肢**:

- A: `0`
- B: `1`
- C: `3`
- D: `2`

**正解**: B (correct_index: 1)

**解説 JA**:
答えは `1` です。これはReactの状態更新とJavaScriptのクロージャの仕組みに関係しています。

`handleClick` が呼ばれた時点で、関数内の `count` はクロージャによって `0` という値をキャプチャしています。そのため、3回の `setCount(count + 1)` はすべて `setCount(0 + 1)` と同じ意味になり、最終的に `count` は `1` になります。

「変数が更新されるはず」と考えがちですが、`count` はレンダリング時に確定した値（スナップショット）であり、`setCount` を呼んでも同じレンダリングサイクル内では変わりません。

もし連続的にインクリメントしたい場合は、関数型更新 `setCount(prev => prev + 1)` を使います。これならReactが最新の値を引数として渡してくれるため、3回呼べば `3` になります。

**解説 EN**:
The answer is `1`. This relates to how React state updates work together with JavaScript closures.

When `handleClick` is called, the `count` variable inside the function is captured by the closure as `0`. Therefore, all three `setCount(count + 1)` calls are equivalent to `setCount(0 + 1)`, and `count` ends up as `1`.

It's tempting to think that `count` gets updated after each `setCount` call, but `count` is a snapshot — a fixed value determined at render time. Calling `setCount` doesn't change it within the same render cycle.

If you want to increment sequentially, use the functional update form: `setCount(prev => prev + 1)`. This way React passes the latest pending state as the argument, and three calls would result in `3`.

## Q2. [code] [medium]

**JA**: 以下のコンポーネントをマウントすると、画面上の `count` の表示はどうなりますか？

```jsx
function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);
    }, 1000);
    return () => clearInterval(id);
  }, []);

  return <div>{count}</div>;
}
```

**EN**: When this component mounts, what will the displayed `count` look like over time?

```jsx
function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);
    }, 1000);
    return () => clearInterval(id);
  }, []);

  return <div>{count}</div>;
}
```

**選択肢**:

- A: 毎秒1ずつ増加する（0 → 1 → 2 → 3 → ...） / Increments by 1 every second (0 → 1 → 2 → 3 → ...)
- B: `1` になった後、ずっと `1` のまま変わらない / Becomes `1` and stays at `1` forever
- C: `0` のまま変わらない / Stays at `0` and never changes
- D: 毎秒再レンダリングが発生し、パフォーマンスが悪化してクラッシュする / Re-renders every second, causing performance degradation and crash

**正解**: B (correct_index: 1)

**解説 JA**:
答えは「`1` になった後、変わらない」です。これはuseEffectにおける「stale closure（古いクロージャ）」という典型的な落とし穴です。

`useEffect` の依存配列が `[]`（空）なので、エフェクトはマウント時に1回だけ実行されます。その時点で `count` は `0` なので、`setInterval` のコールバック内の `count` はずっと `0` をキャプチャしたままです。

つまり、毎秒 `setCount(0 + 1)` が実行されますが、毎回同じ値 `1` をセットしているだけです。最初の1秒後に `0 → 1` に変わりますが、それ以降は `1` のままです（`1` を再度セットしても再レンダリングは発生しません）。

修正方法は、関数型更新を使うことです: `setCount(prev => prev + 1)`。これなら現在の最新の値を参照できるため、期待通りインクリメントされます。

**解説 EN**:
The answer is "becomes `1` and stays there." This is a classic pitfall known as the "stale closure" problem in useEffect.

Since the dependency array is `[]` (empty), the effect runs only once on mount. At that point, `count` is `0`, so the `setInterval` callback permanently captures `count` as `0`.

This means `setCount(0 + 1)` runs every second, but it always sets the same value `1`. After the first second, `count` changes from `0` to `1`, but after that, setting `1` again has no effect (React skips re-renders when the new state equals the current state).

The fix is to use the functional update form: `setCount(prev => prev + 1)`. This reads the latest pending state instead of relying on the closed-over value.

## Q3. [multiple] [medium]

**JA**: 以下のコンポーネントにはバグがあります。何が問題ですか？

```jsx
function UserList({ users }) {
  const sortedUsers = useMemo(
    () => users.sort((a, b) => a.name.localeCompare(b.name)),
    [users]
  );

  return sortedUsers.map(u => <div key={u.id}>{u.name}</div>);
}
```

**EN**: The following component has a bug. What is the problem?

```jsx
function UserList({ users }) {
  const sortedUsers = useMemo(
    () => users.sort((a, b) => a.name.localeCompare(b.name)),
    [users]
  );

  return sortedUsers.map(u => <div key={u.id}>{u.name}</div>);
}
```

**選択肢**:

- A: `useMemo` は不要 — 配列のソートは十分高速なので最適化の意味がない
- B: `users.sort()` は元の配列を破壊的に変更するため、propsのイミュータビリティに違反している
- C: ソート処理は副作用なので `useEffect` 内で行うべきである
- D: `useMemo` はレンダリングをブロックするため、`React.memo` を代わりに使うべきである

**正解**: B (correct_index: 1)

**解説 JA**:
`Array.prototype.sort()` は**破壊的メソッド**で、元の配列を直接変更します。ここでは `users` はpropsとして親コンポーネントから渡されたものなので、それを直接変更するとReactのイミュータビリティの原則に違反します。

具体的にどんな問題が起きるかというと:
- 親コンポーネントが保持している配列が知らないうちに書き換わる
- Reactが変更を検知できず、再レンダリングが正しく行われない可能性がある
- デバッグが困難な予期しない挙動の原因になる

修正するには、スプレッド構文でコピーしてからソートします: `[...users].sort(...)`

選択肢Aについて: `useMemo` が不要かどうかは配列サイズ次第ですが、それはこのコードの**バグ**ではありません。選択肢Cについて: ソートは描画に必要な計算であり副作用ではないので、`useEffect` は不適切です。選択肢Dについて: `useMemo` はレンダリングをブロックしません。`React.memo` はコンポーネントのメモ化で、計算結果のメモ化とは目的が異なります。

**解説 EN**:
`Array.prototype.sort()` is a **mutating method** — it modifies the original array in place. Here, `users` is a prop passed from the parent component, so mutating it violates React's immutability principle.

The concrete problems this causes:
- The parent's array gets silently modified
- React may not detect the change, leading to incorrect re-rendering behavior
- It creates hard-to-debug unexpected behavior

The fix is to copy the array before sorting: `[...users].sort(...)`

Regarding option A: whether `useMemo` is unnecessary depends on the array size, but that's not a **bug**. Option C: sorting is a computation needed for rendering, not a side effect, so `useEffect` is inappropriate. Option D: `useMemo` does not block rendering. `React.memo` memoizes components, which serves a different purpose than memoizing computed values.

## Q4. [truefalse] [easy]

**JA**: 「`useEffect` をif文の中で条件付きで呼び出しても、依存配列を正しく設定していれば問題ない」— これは正しいですか？

**EN**: "It's fine to call `useEffect` inside an if statement conditionally, as long as you set the dependency array correctly." — Is this true?

**正解**: false

**解説 JA**:
これは**誤り**です。Reactのフック（`useState`, `useEffect`, `useMemo` など）は、**すべてのレンダリングで同じ順序・同じ数**で呼ばれなければなりません。これが「Hooksのルール」です。

なぜこのルールがあるのかというと、Reactはフックを**呼び出し順序**で内部的に管理しているからです。名前やキーではなく、「1番目のフック」「2番目のフック」という位置情報で状態を紐付けています。

もし `if` 文の中でフックを呼ぶと、条件によってフックの数や順序が変わり、Reactが状態を正しいフックに紐付けられなくなります。例えば:
- 1回目のレンダリング: `useState` → `useEffect` → `useState`（3つ）
- 2回目のレンダリング: `useState` → `useState`（2つ、`useEffect` がスキップ）
- → Reactは2番目の `useState` を `useEffect` の状態と誤って紐付けてしまう

条件付きでエフェクトを実行したい場合は、**`useEffect` の中で条件分岐**します:
```jsx
useEffect(() => {
  if (condition) { /* ... */ }
}, [condition]);
```

**解説 EN**:
This is **false**. React hooks (`useState`, `useEffect`, `useMemo`, etc.) must be called in the **same order and same number** on every render. This is the "Rules of Hooks."

The reason is that React tracks hooks by their **call order** internally — not by name or key, but by position ("1st hook," "2nd hook," etc.).

If you call a hook inside an `if` statement, the number and order of hooks can change between renders, causing React to associate state with the wrong hook. For example:
- 1st render: `useState` → `useEffect` → `useState` (3 hooks)
- 2nd render: `useState` → `useState` (2 hooks, `useEffect` skipped)
- → React mistakenly maps the 2nd `useState` to `useEffect`'s state

To conditionally run an effect, put the condition **inside** `useEffect`:
```jsx
useEffect(() => {
  if (condition) { /* ... */ }
}, [condition]);
```

## Q5. [explain] [medium]

**JA**: Reactの「Hooksのルール（Rules of Hooks）」とは何か、そしてなぜそのルールが必要なのかを説明してください。

**EN**: Explain the "Rules of Hooks" in React and why these rules are necessary.

**ルーブリック**:
| Score | JA | EN |
|-------|----|----|
| 1 | 「トップレベルで呼ぶ」「条件分岐内で使わない」などルールの一部を言えるが、理由は説明できない | Can state some rules like "call at top level" or "don't use in conditions" but cannot explain why |
| 2 | ルールを正しく列挙でき、「Reactが順序で管理している」程度の理由に触れられる | Can correctly list the rules and mention that "React tracks hooks by order" |
| 3 | ルールと理由を明確に説明し、順序がずれた場合に何が起きるかを具体例で示せる | Clearly explains rules and reasons, with a concrete example of what breaks when order changes |
| 4 | 上記に加え、Reactの内部実装（linked listでの管理）や、カスタムフックでも同じルールが適用される理由まで言及できる | All of the above, plus mentions React's internal implementation (linked list) and why custom hooks follow the same rules |

**模範解答 JA**:
Hooksのルールは主に2つあります。第一に、フックはコンポーネントのトップレベルでのみ呼び出すこと。if文、forループ、ネストされた関数の中では呼び出せません。第二に、フックはReactの関数コンポーネントまたはカスタムフック内でのみ呼び出すこと。

このルールが必要な理由は、Reactがフックを名前ではなく呼び出し順序で管理しているからです。内部的にはlinked listのようなデータ構造で、各レンダリングでN番目に呼ばれたフックをN番目の状態スロットに対応させています。

もし条件分岐でフックをスキップすると、その後のフックの順序がずれて、例えば `useState` の値が別の `useEffect` に紐付けられるといった深刻なバグが発生します。

カスタムフックも内部でフックを呼び出すため、同じルールに従う必要があります。ESLintの `eslint-plugin-react-hooks` を使うと、このルール違反をビルド時に検出できます。

**模範解答 EN**:
There are two main Rules of Hooks. First, hooks must be called at the top level of a component — not inside if statements, loops, or nested functions. Second, hooks must only be called from React function components or custom hooks.

These rules exist because React tracks hooks by their call order, not by name. Internally, React uses a linked list-like structure where the Nth hook called during a render corresponds to the Nth state slot.

If a hook is conditionally skipped, all subsequent hooks shift position, causing state to be mapped to the wrong hook — for example, a `useState` value being associated with a `useEffect`. This leads to serious bugs.

Custom hooks must follow the same rules because they call hooks internally. The `eslint-plugin-react-hooks` ESLint plugin can catch violations of these rules at build time.
